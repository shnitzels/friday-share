השורה התחתונה

Code with Claude 2026 הוא לא עוד אירוע “המודל החדש יותר חכם”. זו הצגה של עולם שבו Claude הופך משכבת תשובות לשכבת ביצוע: סוכנים שרצים שעות, עובדים על קוד ועסקים, משתמשים בזיכרון, מפעילים כלים, נמדדים ב-evals, ומתחברים לתשתיות ארגוניות אמיתיות.

  • המעבר המרכזי: מצ׳אט או coding assistant אל agentic infrastructure.
  • הבוטלנק החדש: לא רק intelligence, אלא verification, permissions, observability, context, cost, latency ו-governance.
  • ה-moat הארגוני: הידע הפנימי: data, workflows, runbooks, skills, policies, tools.
  • הפרימיטיבים החדשים: Managed Agents, memory, dreaming, MCP tunnels, self-hosted sandboxes, outcomes, effort levels ו-prompt caching.
  • הלקח למוצרי AI: מי שבונה רק “UI יפה מעל מודל” נשאר רדוד. מי שבונה מערכת שמחזיקה context, בודקת את עצמה, יודעת לפעול ומתחברת לארגון, מתחיל לייצר יתרון אמיתי.

  • Claude Code כבר משמש לקנה מידה אמיתי: Spotify משתמשת בו למיגרציות של אלפי repos, עם agents שפותחים PRs.

  • ההמלצה למפתחים: לבנות לא ליכולות של היום, אלא ליכולות שיגיעו. ארכיטקטורה צריכה לספוג קפיצות מודל.
  • Scaffolding שהועיל פעם עלול להפוך לחסם. מודלים חכמים יותר עובדים טוב עם פרימיטיבים כלליים כמו filesystem ו-sandbox.
  • Managed Agents מוצגים כשכבת תשתית: sessions, סביבות עבודה, event streams, sandboxes, זיכרון ויכולת להריץ agents בקנה מידה.
  • Self-hosted sandboxes ו-MCP tunnels הם ניסיון לפתור את החיבור לפרודקשן: להריץ כלים בסביבה שלכם ולחבר MCP פנימי בלי לחשוף אותו החוצה.
  • Growthbot בדמו מתנהג כסוכן פרואקטיבי: מזהה בעיות onboarding, מציע next steps ומשתף פעולה עם בני אדם.
  • Claude Code מקבל יכולות כמו עבודה ברקע, schedule, ו-routines, כלומר פחות “תכתוב לי קוד” ויותר “תחזיק תהליך”.

3. Running an AI-native engineering org

סרטון מקור · תמלול זמין

  • כשה-agentic coding הופך לברירת מחדל, הבעיה אינה “איך גורמים למהנדסים להשתמש בכלי”. הבעיה היא שהתהליכים הישנים נשברים.
  • ב-Anthropic הבוטלנקים עברו ל-verification, review, ownership והחלטות מוצריות/ארגוניות.
  • Refactors, cleanup וארכיטקטורה שהיו נדחים כי “אין זמן” פתאום נהיים אפשריים. לכן צריך להחליט מחדש מה ראוי לעשות, ומה כבר לא חשוב.
  • הארגון צריך shift-left ל-verification: לא לחכות לסוף, אלא לבנות איכות, בדיקות ו-review לתוך הזרימה.
  • Claude Code review מוזכר ככלי שעוזר, אבל עדיין צריך איזון בין אוטומציה לבין שיקול דעת אנושי.
  • לא רק מהנדסים משתנים: גם designers, managers וצוותי מוצר יכולים לעבוד בתוך זרימות Claude Code כשהכלי מחובר לתהליכים שלהם.
  • אחד הכללים הארגוניים החשובים: לתת הרשאה מפורשת להרוג תהליכים ישנים. אם AI מאיץ עבודה אבל הארגון משאיר את כל הטקסים הישנים, הערך נבלע.
  • שאלה שחוזרת: “אם אני עושה את זה, האם Claude יכול לעשות את זה בשבילי כדי לפנות אותי לבעיה חשובה יותר?”

5. The prompting playbook: Prompting הוא הנדסה, לא קסם

סרטון מקור · תמלול זמין

  • הדוברת מציגה prompting דרך דוגמת בוט שירות לקוחות של Meridian Mobile, עם eval suite של חמישה מקרי בדיקה.
  • המהלך הראשון הוא ניקוי prompt: להסיר בלבול, להבהיר תפקיד, פורמט, מדיניות וטון. זה לבד משפר ביצועים.
  • אבל הוראות לא מוסיפות capability. להגיד למודל “תחשב נכון, זה קריטי” לא הופך אותו לטוב במתמטיקה.
  • כשהבעיה היא יכולת חישוב או פעולה, הפתרון הוא tool. בדוגמה של proration, נותנים למודל כלי חישוב ולא מצפים ממנו לחשב בראש.
  • Prompt טוב נבנה עם evals. אחרת אין דרך לדעת אם שיפור אחד שבר מקרה אחר.
  • במשימות agentic צריך לעצב לולאה, לא רק prompt אחד: generate, evaluate, repair.
  • בדוגמת retail staff schedule, בדיקה תכנותית סופרת violations בתוצר. כך הופכים איכות למשהו מדיד.
  • מודל חזק יותר כמו Opus 4.7 יכול לעזור, אבל גם אז הלקח אינו “תמיד קח מודל יקר”. הלקח הוא לבדוק מול הבעיה.
  • הפלייבוק: להתחיל פשוט, למצוא כשלים, להוסיף כלים או rubrics כשצריך, למדוד, ואז לשפר.

9. Building signals that trade themselves: Skills כ-governance לפרודקשן

סרטון מקור · תמלול זמין

  • Man Group מנהלת מעל 200 מיליארד דולר, ולכן AI בפרודקשן הוא לא ניסוי חמוד. טעות עולה כסף אמיתי.
  • הדוברת מתארת signals למסחר שרצים עם הון אמיתי, וש-AI היה מעורב ברעיון, data discovery, backtest, proposal ואף productionization.
  • הסוד אינו רק מודל חזק, אלא חיבור Claude לידע הארגוני: data, תשתיות, תהליכים, שיטות מחקר ו-governance.
  • Skills הופכים לשכבת ההקשר הארגונית. הם מלמדים את Claude איך Man Group עובדת בלי fine-tuning.
  • אבל skills בלי משטר יוצרים local optimizations מסוכנים. הדוגמה: skill לדוחות הוצאות עם cost center hardcoded ששלח אישורים למנהל הלא נכון.
  • הפתרון: marketplace ארגוני ל-skills, עם owners, review, evals, tags, usage tracking ו-lifecycle.
  • המסר: skills צריכים יחס של production code, כי הם מפעילים workflows אמיתיים.
  • בדמו, Claude מחפש data אלטרנטיבי, בוחן הוצאות כרטיסי אשראי מול תשואות Amazon, מריץ backtest ומרחיב את הבדיקה ל-retail.
  • הנתונים המרשימים: כ-750 משתמשי Claude Code בארגון, מעל 100 governed skills, וחזון של swarms of agents שמחפשים הזדמנויות השקעה.

10. Picking the right model: לבחור לפי outcome, לא לפי benchmark

סרטון מקור · תמלול זמין

  • החלוקה הפשוטה “Opus לחכם, Haiku לזול, Sonnet באמצע” לא מספיקה.
  • Benchmarks ציבוריים כמו SWE-bench או BrowseComp הם כיוון בלבד. הם לא מייצגים את המוצר שלך.
  • צריך eval פרטי שמייצג משימות אמיתיות, inputs אמיתיים, success criteria וכלים אמיתיים.
  • ב-agentic workflows חשוב לבדוק גם את הדרך, לא רק את התשובה הסופית.
  • Graders יכולים להיות LLM-as-judge וגם בדיקות דטרמיניסטיות בקוד.
  • מדדים יכולים לשקר: infra failures אינם model failures, ו-dataset רווי כבר לא מודד התקדמות.
  • המדד הנכון הוא cost per successful outcome, לא מחיר פר token.
  • Prompt caching ו-context engineering יכולים להזיז את כל העקומה: איכות דומה בעלות נמוכה יותר.
  • דוגמאות: cache hit rate של 80-90% כמטרה; ניקוי tool responses יכול להפחית tokens בעשרות אחוזים ואף לשפר accuracy.

11. Memory and dreaming: סוכנים שלומדים בין משימות

סרטון מקור · תמלול זמין

  • בלי זיכרון, כל session מתחיל מאפס. עם זיכרון, agent אמור להשתפר ממשימה למשימה.
  • Anthropic מממשת memory כ-filesystem כי Claude טוב בניווט קבצים, bash ו-grep. “Get out of Claude’s way”.
  • Memory מיועד גם ל-multi-agent: stores משותפים, scopes של read/write, היררכיות, diffs, attribution ו-audit trail.
  • Dreaming הוא batch process מחוץ ל-hot path: הוא מנתח transcripts, מסדר זיכרון, מזהה כפילויות ודפוסי טעויות, ומעדכן את המערכת.
  • הדמו מציג SRE agents שמטפלים באלרטים, משתפים מידע, ו-dream מזהה דפוס שחוזר בין sessions.
  • הדוגמאות שהוזכרו: Rakuten עם ירידה של 97% ב-first-pass errors, WiseDocs עם cross-session memory, Harvey עם פי 6 completion rate ב-legal benchmark עם dreaming.

12. Legal agents inherit from coding agents: Legora

סרטון מקור · אין תמלול, מבוסס על תיאור רשמי בלבד

לפי התיאור, Legora בנתה legal-AI agent סביב שלושה דפוסים מעולם coding agents: מה אפשר למחזר, מה צריך לתרגם, ומה צריך להמציא מחדש. הדוגמאות הן document editing, linting ו-bulk review.

הנקודה החשובה: coding agents הם לא רק כלי לקוד. הם blueprint לעבודה knowledge-heavy עם מסמכים, כללים, בדיקות, עריכה וריוויו.

13-14. Claude Managed Agents: להפוך infra של agents לקומודיטי

How to get to production faster, Build a production-ready agent · תמלול זמין

  • הטענה המרכזית: הבוטלנק עבר מהמודל לתשתית. צריך sessions, environments, tools, event streams, orchestration, monitoring ו-permissions.
  • Session הוא event log חי: user events, agent events, lifecycle events ו-span events.
  • User events כוללים טקסט, תמונות, מסמכים, interrupts, tool results, confirmations ו-outcome definitions.
  • Outcomes מאפשרים להגדיר rubric ולתת ל-Claude לעשות first pass, לבדוק את עצמו, ולהמשיך iteration.
  • Self-hosted sandboxes ו-MCP tunnels פותרים את הקשר לסביבות פרטיות: Cloudflare, Daytona, Modal, Vercel או תשתית שלכם.
  • בדמו Pascal, agent מנתח grocery shopping habits, מציג dashboard, ומאפשר צפייה ב-event stream וב-tool runs.
  • בסדנת build, ה-agent מורכב מ-Agent, Environment, Session ו-Events. אפשר preload לקבצים, repos ו-resources.
  • דמו Deal Desk ל-M&A משתמש ב-Linear MCP, memory ו-multi-agent orchestration כדי להשוות חברות פיקטיביות.
  • Credential vaults מאפשרים ל-Claude להשתמש בהרשאות בלי להכניס tokens ל-context window.
  • המסר למפתחים: אל תבנו שוב agent loop, sandbox fleet, durable state ו-auth אם הפלטפורמה יכולה לתת את זה כפרימיטיב.

15. The thinking lever: כמה Claude צריך לחשוב?

סרטון מקור · תמלול זמין

  • Test-time compute הוא מנוף איכות: יותר reasoning tokens יכולים לשפר ביצועים, אבל עולים כסף וזמן.
  • Effort level הוא tradeoff בין quality, latency ו-cost.
  • Thinking אינו פעולה אחת: יש thinking, tool calling ו-text output.
  • Adaptive thinking מאפשר למודל לבחור מתי לחשוב, מתי לקרוא לכלי ומתי לענות.
  • כיבוי thinking הוא פרוקסי גרוע ל-effort. עדיף להשאיר capability ולשלוט ברמת מאמץ.
  • הדרך לבחור effort היא evals: להריץ מודלים ורמות effort שונות על משימות קשות ורלוונטיות.
  • יש diminishing returns: max effort יכול להיות טוב יותר, אבל לא תמיד שווה את העלות.
  • ברירת המחדל המומלצת כשלא בטוחים: extra high, האיזון של Claude Code ו-Claude.ai.
  • למשימות פשוטות כמו classification, summarization או extraction אפשר לרדת ל-low/medium.

מה זה אומר לסטארטאפים ולמוצרי AI

  • UX כבר לא מספיק. מוצר agentic טוב צריך זיכרון, כלים, הרשאות, traceability ויכולת להמשיך עבודה לאורך זמן.
  • Design-aware ו-context-aware הם אותו קרב. מי שמבין את מערכת העבודה של המשתמש, הקבצים, המותג, ה-tools וה-rubrics שלו, מייצר תוצאה שקשה להחליף.
  • Skills הם ה-API החדש של ידע ארגוני. לא עוד promptים מפוזרים, אלא assets מנוהלים עם ownership ו-evals.
  • המודל הנכון הוא החלטה תפעולית. צריך לבחור לפי successful outcome, effort, latency, cache ו-context hygiene.
  • ארגונים יצטרכו להרוג תהליכים. אם AI מאיץ כתיבה, בדיקה ובנייה, אבל הארגון שומר את כל הוועדות והטקסים, הוא לא מקבל 10x. הוא מקבל עומס.
