The Prompting Playbook: איך מתקנים prompts קיימים ובונים סוכנים מאפס

מרגו ונלאר מ־Anthropic מציגה פלייבוק מעשי לפרומפטינג: להתחיל מ־evals, לנקות prompt עמוס, לבודד failure modes, להימנע מטלאים ישנים, ולפעמים לפרק משימה אייג׳נטית ללולאת generate, evaluate, repair במקום להעמיס הכל על prompt אחד.

מקור: The prompting playbook · תמלול זמין

השורה התחתונה

ההרצאה על The Prompting Playbook מחזירה את הפרומפטינג למקום הנכון שלו: לא קסם מילולי ולא רשימת טריקים, אלא דיסציפלינה הנדסית. מרגו ונלאר, applied AI engineer ב־Anthropic, מציגה שתי סיטואציות שמפתחים פוגשים כל הזמן. הראשונה היא prompt קיים בפרודקשן, שנבנה עם הזמן על ידי כמה אנשים, צבר טלאים, ועכשיו מתנהג אחרת בגלל מעבר למודל חדש או שינוי ארכיטקטורה. השנייה היא בניית use case אייג׳נטי חדש מאפס.

הטענה המרכזית היא שלפני שמתקנים prompt צריך לדעת איך מודדים הצלחה. לכן נקודת הפתיחה היא evals: סט בדיקות שמכסה גם מקרה בקרה שאמור תמיד לעבור, גם מקרים שבהם רוצים לזהות רגרסיה, וגם מקרים שמייצגים התנהגות מורכבת או גבולית. בלי evals, שינוי prompt הוא ניחוש. עם evals, אפשר לראות אם ניקוי, מבנה, או שינוי הוראה באמת שיפרו את המערכת. זה גם מאפשר לנהל דיון צוותי ענייני: לא ״הפרומפט מרגיש טוב יותר״, אלא אילו מקרי בדיקה עברו, אילו נכשלו ומה המחיר ב־tokens ולטנטיות.

תרחיש ראשון: prompt קיים שנשבר

הדוגמה בהרצאה היא בוט שירות לקוחות לחברת טלקום. ה־prompt כולל תפקיד, נתוני לקוח, מדיניות, הנחיות טון וכללי חישוב, אבל כמו prompts רבים בפרודקשן הוא התרחב בלי בעלים ברור. יש בו טקסט שהועתק מאתר, אזכורים לא רלוונטיים כמו hero image וקוקיז, הוראות שמציגות את הבוט כאדם, ובלוקים ארוכים שבהם מדיניות, טון ותהליך מעורבבים יחד.

השלב הראשון הוא היגיינה כללית: להסיר מידע מיותר, לתקן הצגות שווא, להוסיף מבנה ולחלק את ההנחיות לאזורים ברורים. בהרצאה מוצג שימוש בתגיות בסגנון XML כדי להפריד בין role, guidelines, policy וטון. זו אינה רק אסתטיקה. כאשר מודל מקבל הוראות מובנות, קל יותר להבין מה כלל מחייב, מה הקשר ומה סגנון.

בודדים failure modes במקום לירות לכל הכיוונים

אחרי הניקוי, ונלאר עוברת לתיקון ממוקד של כשלונות. במקרה הראשון, לקוח שואל כמה נתוני hotspot יש לו בתוכנית unlimited. הלקוח נמצא בתוכנית legacy, והנתון האמיתי קיים בפרטי החשבון: 5GB. אבל ה־prompt מכיל טלאי ישן שמזהיר לא לתת פרטי תוכנית שגויים ולהפנות ל־URL במקום זאת. מודל חדש, שמציית טוב יותר להוראות, מתלבש על האזהרה ומתחמק מתשובה גם כשיש לו את הנתון המדויק. הלקח: הוראות שנכתבו כדי לפצות על חולשה של מודל קודם עלולות להפוך לרגרסיה במודל חדש.

במקרה נוסף, הבוט צריך להסלים לטיפול אנושי בגלל קונפליקט חיוב. במקום זאת הוא מנסה להסביר ולפתור לבד, משום שה־prompt מדגיש שהעברת שיחה לנציג עולה שמונה דולרים ופוגעת במדדי הצוות. הבעיה היא שהמודל קיבל רק צד אחד של פונקציית העלות. התיקון הוא לתת גם את הצד השני: אם הוא טועה, המחיר הוא החזר כספי ופגיעה באמון הלקוח. זה שיעור חשוב: מודלים אופטימיזטוריים מגיבים למטרות שמנסחים להם, ואם מנסחים מטרה חלקית הם יתאימו אליה יותר מדי.

תרחיש שני: בונים סוכן חדש מאפס

בחלק השני ההרצאה עוברת לשימוש אייג׳נטי חדש: יצירת לוחות זמנים לפי אילוצים. כאן השאלה היא לא רק איך לנסח טוב יותר, אלא האם בכלל נכון לבקש ממודל אחד לפתור הכל בבת אחת. ונלאר מראה ניסיונות שונים: שימוש במודל חזק יותר עם adaptive thinking מצליח יותר, אבל צורך הרבה יותר tokens ומעלה latency. שימוש במודל קטן יותר עם prompt משופר עוזר, אך עדיין נתקל במגבלות פלט וזמן.

הפתרון המעניין הוא לפרק את העבודה ללולאת generate, evaluate, repair. prompt אחד מייצר טיוטת לוח זמנים, prompt שני בודק הפרות של כללים ומביא ראיות, ו־prompt שלישי מבצע תיקונים ממוקדים. במקום prompt ענק שמנסה לזכור את כל התפקידים, המערכת מחלקת משימה חוזרת לשלבים ברורים. לפי הדמו, הגישה הזו פותרת את כל מקרי הבדיקה עם פחות tokens ולטנטיות נמוכה יותר מאשר ניסיון לדחוף הכל למודל אחד.

יש לכך יתרון נוסף: אפשר להכניס אילוצים רכים בזמן ריצה. למשל, עובד מסוים לא מסתדר עם עובד אחר, או שצריך משמרת נוספת ביום מסוים. במקום לשנות פונקציית בדיקה קשיחה בכל פעם, אפשר להכניס את האילוץ לשלב ההערכה או התיקון ולתת למערכת להתאים את עצמה.

המשמעות למפתחים ולסטארטאפים

למפתחים, הפלייבוק הוא מתודולוגיה: בנו evals, נקו prompt, תקנו failure modes אחד אחד, הסירו טלאים ישנים, נסחו tradeoffs מלאים, ובדקו אם משימה מורכבת צריכה פירוק אייג׳נטי. לסטארטאפים, זה ההבדל בין דמו שעובד על חמישה מקרים לבין מוצר שאפשר לתחזק. prompt בפרודקשן הוא נכס תוכנה: צריך ownership, בדיקות רגרסיה, דיפ, code review ושפה משותפת.

נקודות לקחת הלאה

  • בלי evals, אי אפשר לדעת אם שינוי prompt שיפר או הזיק.
  • היגיינה בסיסית, מבנה והסרת מידע מיותר יכולים לשפר ביצועים מיד.
  • טלאים שנכתבו למודל ישן עלולים להכשיל מודל חדש שמציית טוב יותר.
  • כשנותנים למודל עלות, צריך לתת גם את התועלת והסיכון שמולה.
  • במשימות מורכבות, לולאת generate, evaluate, repair יכולה להיות יעילה יותר מ־prompt אחד גדול.
