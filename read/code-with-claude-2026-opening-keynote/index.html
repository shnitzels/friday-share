נאום הפתיחה של Code with Claude London 2026: מהקסם של התכנות אל ארגונים אייג׳נטיים

הכנס בלונדון מציג את התזה המרכזית של Anthropic: המודלים משתפרים בקצב אקספוננציאלי, והיתרון יעבור למפתחים ולחברות שיבנו כבר עכשיו תהליכים, תשתיות ואימותים שמתאימים לעבודה אסינכרונית עם סוכני Claude.

מקור: Code with Claude London 2026: Opening Keynote · תמלול זמין

השורה התחתונה

נאום הפתיחה של Code with Claude London 2026 הוא פחות רשימת הכרזות ויותר הצהרת כיוון: התכנות חוזר להיות קרוב יותר לתחושת הקסם המקורית שלו, אבל בקנה מידה ארגוני. הדובר פותח בזיכרונות מתכנות מחשבון מדעי ו־HTML למודעות איביי, כדי להזכיר שמפתחים תמיד רצו לגרום למחשב לעשות משהו שימושי במהירות. ההבדל הוא שכיום הקסם הזה עובר דרך מודלים שמסוגלים לעבוד שעות, ובהמשך צפויים לעבוד ברציפות, לנהל מטרות רחבות ולשתף פעולה עם בני אדם וסוכנים אחרים.

המסר המעשי למפתחים ברור: אל תבנו רק למה שהמודל מסוגל לעשות היום. בנו ארכיטקטורות שמוכנות לקפיצה הבאה. ככל שיכולות המודל גדלות, חלק מהסקפולדינג הישן, כלומר הלולאות, ההוראות והכלים שנועדו לפצות על חולשות המודל, עלול דווקא להגביל אותו. לכן יש חשיבות גוברת לפרימיטיבים כלליים, לסביבות עבודה בטוחות, לאימות אוטומטי ולהערכות קשות שמגלות מתי משימה שבעבר נכשלה פתאום הופכת אפשרית. זהו מעבר ממחשבה על מודל כעל עוזר נקודתי למחשבה עליו כעל שכבת ביצוע מתפתחת, שצריך לאפשר לה מרחב פעולה בלי לוותר על בקרה.

מה הוצג בנאום

הנאום מסדר את היום סביב שלוש שכבות: שכבת המודלים, שכבת הפלטפורמה ושכבת העבודה היומיומית של מפתחים. ברמת המודלים, Anthropic מציגה את הרעיון של אופק משימה: כמה זמן מודל יכול לעבוד לפני שהוא מאבד את החוט. אם בעבר דיברו על דקות, היום מדובר בשעות, והחזון הוא סוכנים רציפים, פרואקטיביים, שמקבלים אחריות על יעדים כמו שמירת פרויקט במסלול או תחזוקת תחזית פיננסית לאורך זמן. הדגש אינו על תשובה בודדת, אלא על אחריות מתמשכת שמחייבת זיכרון, שיפוט ושיתוף פעולה.

ברמת הפלטפורמה, הדגש הוא על Claude Managed Agents: תשתית שמטרתה לקצר את הדרך מסוכן ניסיוני למערכת פרודקשן. הוצגו יכולות כמו אורקסטרציה של כמה סוכנים, הגדרת תוצאות רצויות, Dreaming שמאפשר לסוכן להפיק לקחים מתמלילים קודמים, וכן הכרזות על self-hosted sandboxes ועל MCP tunnels. המשמעות היא שסוכנים יכולים לבצע עבודה בסביבה שהארגון שולט בה, וגם לגשת בצורה מאובטחת לשרתי MCP פנימיים ולמערכות ידע ארגוניות.

הדמו המרכזי בפלטפורמה עסק ב־Growthbot לחברת דמה בשם Counter. הסוכן מזהה ניסוי מוצרי שבו גרסת אונבורדינג פשוטה מנצחת גרסה ארוכה יותר, מחשב השפעה על הכנסות באמצעות גישה מאובטחת למחסן הנתונים, ומציע להתחיל בניקוי הווריאנט הישן. זה דמו שמראה את ההבדל בין צ׳אטבוט שנותן תשובה לבין סוכן שמחזיק הקשר עסקי, מפעיל כלים, מבין תוצאה רצויה ומבקש אישור לפעולה הבאה.

Claude Code והמעבר לעבודה אסינכרונית

בחלק האחרון עוברים ל־Claude Code ולפער שבין יכולות המודל לבין שגרת המפתח. כאן מוצגות יכולות שמטרתן להפוך קידוד אסינכרוני לברירת מחדל: אפליקציית דסקטופ שמרכזת סשנים רבים, שליטה מרחוק דרך iOS ו־Android, Autofix שמטפל בתגובות קוד ריוויו, CI, קונפליקטים ומיזוגים, ו־Routines שמריצות משימות לפי לוח זמנים, webhook או קריאת API.

בדמו, Claude Code מוסיף refunds לדשבורד סוחרים של Acme Pay, כולל idempotency, תמיכה בריבוי מטבעות ולוגים לצוות compliance. חשוב יותר מהיישום עצמו הוא תהליך האימות: Claude מפעיל את הדשבורד, מזהה כשל אמיתי בדפדפן, מאתר race condition בעדכון אופטימי, מתקן ומוודא שהבעיה נפתרה. לאחר מכן מוצגת תמונה רחבה יותר: לא סשן אחד של קידוד, אלא כמה סשנים שמתנהלים במקביל, חלקם ממתינים לקלט וחלקם כבר פתחו או מיזגו Pull Requests.

המשמעות למפתחים ולסטארטאפים

לסטארטאפים, ההכרזה החשובה אינה פיצ׳ר אחד אלא שינוי תפעולי. אם בעבר היתרון היה מי כותב קוד מהר יותר, עכשיו היתרון הוא מי בונה מערכת שמייצרת, בודקת, מתקנת וממזגת עבודה באופן אמין. צוות קטן יכול להרחיב תפוקה, אבל רק אם הוא משקיע באימותים, CI יציב, הרשאות, סביבות sandbox ונתיבי אישור ברורים. בלי זה, התפוקה תהפוך לרעש.

יש כאן גם אזהרה אסטרטגית: חברות שלא יפתחו שריר של הערכה מתמשכת יתקשו לזהות מתי המודל החדש כבר מסוגל לפתור בעיה שנחשבה בלתי אפשרית. הנאום חוזר על הרעיון שהאקספוננט נע מתחת לרגליים. לכן roadmap מוצרי צריך לכלול ניסויים חוזרים, משימות קשות שנשמרות כמדד, ויכולת לשחרר פיצ׳רים ברגע שהיכולת מבשילה. מי שיחכה להוכחה מושלמת מהשוק עלול לגלות שמתחרה עם תשתית מוכנה כבר הטמיע את הקפיצה.

למפתחים, הדיבור על Routines כעל higher order prompt הוא נקודת מפתח. במקום לכתוב prompt ידני בכל פעם, המפתח מגדיר תהליך שמייצר prompts ומריץ אותם בהקשר הנכון. זו שכבת אוטומציה חדשה מעל הקידוד: לא רק לבקש מ־Claude לכתוב פונקציה, אלא להגדיר שומר ריפו שמזהה issue, פותח עבודה, בודק אותה ומטפל בתקלות עד שה־PR ירוק.

נקודות לקחת הלאה

  • בנו למודל הבא, לא רק למודל הנוכחי.
  • החליפו סקפולדינג מסורבל בפרימיטיבים כלליים ואימותים חזקים.
  • סוכנים שימושיים באמת צריכים גישה מאובטחת לכלים, נתונים וסביבות הרצה.
  • קידוד אסינכרוני מחייב תרבות של בדיקות, CI ותצפיתיות.
  • היתרון התחרותי עובר מהיכולת לכתוב קוד ליכולת להפעיל מערכות שמביאות עבודה איכותית עד פרודקשן.
