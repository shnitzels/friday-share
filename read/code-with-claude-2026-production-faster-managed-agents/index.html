להגיע מהר יותר לפרודקשן עם Claude Managed Agents

ההרצאה מציגה את Claude Managed Agents כשכבת תשתית לסוכנים בפרודקשן: סשנים, אירועים, סביבות, orchestration, זיכרון, sandboxes עצמיים, MCP tunnels ושיחה עם שותפי תשתית על מה שנדרש כדי להריץ סוכנים בקנה מידה.

מקור: How to get to production faster with Claude Managed Agents · תמלול זמין

השורה התחתונה

הטענה המרכזית בהרצאה היא שהמודלים כבר חזקים מספיק כדי לבצע משימות ארוכות ומורכבות, אבל הדרך לפרודקשן נתקעת בתשתית: איפה מריצים את הסוכן, איך מנהלים מצב, איך נותנים לו כלים בלי לחשוף סודות, איך עוקבים אחרי מה שהוא עושה, ואיך מאפשרים לו לעבוד עם סוכנים אחרים. Claude Managed Agents מוצג כמענה לשכבה הזו. במקום שכל צוות יבנה agent loop, מערכת אירועים, sandboxing, זיכרון, הרשאות ואחסון durable מאפס, Anthropic מציעה primitive-ים מנוהלים שעליהם אפשר לבנות מוצר.

מה הבעיה ש-Managed Agents מנסה לפתור

הדוברים, מייקל והריסון מאנתרופיק, מתחילים בתיאור קפיצת היכולת של המודלים. בעבר אפשר היה לבקש משימות קצרות יחסית; כיום מודלים כמו Opus מסוגלים לעבוד לאורך זמן, להשתמש בכלים, לכתוב קוד, לנתח נתונים ולהריץ תהליכים מורכבים. לכן, לדבריהם, צוואר הבקבוק כבר פחות “האם המודל מספיק חכם” ויותר “האם יש לו תשתית בטוחה ואמינה לפעול בה”. סוכן בפרודקשן צריך סביבת הרצה, גישה מבוקרת לכלים, זיכרון, מנגנון אירועים, ניטור, התאוששות משגיאות ויכולת לעבוד ברקע.

Managed Agents מחלק את העולם לכמה רכיבים: הגדרת agent עם system prompt, מודל וכלים; environment שבו מוגדרים קונטיינרים, חבילות, רשת ו-MCP; session שהוא השיחה או המשימה המתמשכת; ו-event stream שמאפשר לראות ולהזין אירועים בזמן אמת. במקום לחשוב על סוכן כקריאת API חד-פעמית, ההרצאה מציגה אותו כישות תפעולית עם מחזור חיים.

זרם האירועים: לא רק תשובות טקסט

אחד החלקים החשובים בהרצאה הוא פירוק האירועים. יש user events: הודעות טקסט, תמונות, מסמכים, interruptions, תוצאות של כלים חיצוניים ואישורי human-in-the-loop. יש agent events: תגובות של Claude, הרצת כלים, ותיאום עם סוכנים אחרים. יש session events: שינויי סטטוס, התאוששות משגיאות ועיבוד outcomes. ויש span events, שמסמנים התחלה וסיום של פעולות ארוכות. המבנה הזה משמעותי כי הוא מאפשר למוצר להציג למשתמש מה קורה, להתערב כשצריך, ולנתח בדיעבד איפה נוצרה בעיה.

בדמו הקצר מוצג סוכן בשם Pascal לניתוח הרגלי קניות. המשתמש מפעיל ניתוח, הקונסול מציג את האירועים בזמן אמת, והסוכן מחזיר תובנות על פופולריות מוצרים וזמני עומס. לאחר מכן מופעלת יכולת שמנתחת את תמליל הסשן ומציעה איך לשפר את הקונפיגורציה, למשל לזהות סקריפט Python איטי. זו הדגמה קטנה אך חשובה: observability אינה מותרות אלא חלק מהעיצוב של סוכן שאמור לשרת משתמשים אמיתיים.

יכולות מתקדמות: multi-agent, outcomes, memory, dreaming

ההרצאה סוקרת כמה יכולות מתקדמות שהופכות את Managed Agents ליותר ממעטפת הרצה. Multi-agent orchestration מאפשר ל-Claude לפתוח תתי-סוכנים עם חלונות הקשר משלהם ולהאציל להם עבודה. Outcomes מאפשרים למפתח להגדיר מטרות או rubric, כך שהסוכן יבדוק את עצמו ויחזור על העבודה עד שהוא מרגיש שהתוצאה עומדת בקריטריונים. Memory מאפשר לסוכן לקרוא ולכתוב ממאגרי זיכרון ארוכי טווח כדי להשתפר מסשן לסשן. Dreaming, שהוכרז כ-research preview, מאפשר לסכם ולתקן זיכרונות על בסיס אלפי סשנים, כלומר להפוך ניסיון מצטבר להכוונה שימושית יותר.

Self-hosted sandboxes ו-MCP tunnels

שתי הכרזות מרכזיות בהרצאה נוגעות לארגונים שצריכים לשמור נתונים ותשתיות בתוך גבולותיהם. Self-hosted sandboxes מאפשרים להביא תשתית compute משלכם או של שותפים כמו Cloudflare, Daytona, Modal ו-Vercel. Anthropic שולחת אות לכך שנדרש sandbox, אבל הארגון שולט בסביבה, ברשת, בחבילות, בקבצים ובמדיניות. MCP tunnels מאפשרים לחשוף ל-Claude MCP servers פרטיים דרך תעלה מאובטחת, בלי לפרסם אותם לאינטרנט הפתוח. עבור חברות עם נתונים פנימיים, זה פותר בעיה אמיתית: לתת לסוכן גישה לעולם הארגוני בלי לפרוץ את ההיקף האבטחתי.

מה אמרו שותפי התשתית

בחלק הפאנל, נציגי Cloudflare, Daytona, Modal ו-Vercel מסבירים שכל אחד רואה את הבעיה מזווית אחרת. Vercel מדגישה תשתית אחידה שמאפשרת לפונקציות, builds ו-sandboxes לעבוד על אותו בסיס. Modal מדברת על scale מהיר, GPUs ויכולת להריץ מאות אלפי sandboxes בדקות. Daytona מדגישה שסוכנים צריכים מה שבני אדם צריכים: מערכות הפעלה, CPU, זיכרון, GPU, pause, resume ו-fork. Cloudflare מציגה שני סוגי sandboxing: microVM מלא למקרים שבהם הסוכן פועל כמפתח, ו-isolates קלים להפעלה בקנה מידה עצום.

הפאנל גם מחדד את הבעיות שעדיין פתוחות: זהות והרשאות לסוכנים, סינון egress, בידוד טוב יותר בין agent SDK לבין code execution, אחסון מתמשך, resume-ability, והמעבר מעבודה סינכרונית של אדם יחיד למשחק “מרובה משתתפים” שבו סוכנים פועלים ברקע. זהו חלק חשוב בהרצאה כי הוא מציב את Managed Agents לא כמוצר סגור שפתר הכול, אלא כחלק מאקו-סיסטם חדש של תשתיות סוכנים.

המשמעות למפתחים ולסטארטאפים

עבור מפתחים, ההמלצה המעשית היא לחשוב על סוכן כעל מערכת הפעלה קטנה: יש לו identity, סביבת עבודה, כלים, אירועים, זיכרון ומדדי הצלחה. עבור סטארטאפים, Managed Agents עשוי לקצר מאוד את הדרך למוצר: במקום להשקיע חודשים בתשתית סוכנים, אפשר לבנות סביב workflow ייחודי, נתונים ייחודיים וחוויית משתמש. אבל ההרצאה גם מזהירה בעקיפין: כשהסוכן מקבל יותר כוח, שאלות אבטחה, תצפיתיות והרשאות הופכות למרכזיות.

נקודות לקחת

  • המודלים מתקדמים מהר, אך תשתית פרודקשן היא צוואר הבקבוק.
  • Managed Agents מספקים primitive-ים: agents, environments, sessions ו-event streams.
  • Observability, interrupts ו-human-in-the-loop הם חלק מהותי ממוצר סוכני.
  • Outcomes, זיכרון, Dreaming ותזמור מרובה סוכנים מאפשרים עבודה ארוכה ואיכותית יותר.
  • Self-hosted sandboxes ו-MCP tunnels מיועדים לארגונים שצריכים שליטה ברשת, בהרשאות ובנתונים.
