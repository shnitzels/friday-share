לבנות סוכן פרודקשן עם Claude Managed Agents

סשן hands-on שמראה איך עוברים מתיאוריה לקוד: בניית אפליקציית Deal Desk עם סשנים, אירועים, streaming, multi-agent, outcomes, memory stores וניטור חי בקונסול של Claude.

מקור: Build a production-ready agent with Claude Managed Agents · תמלול זמין

השורה התחתונה

הסרטון הזה הוא סדנת בנייה, לא רק הרצאת מוצר. הדובר פותח בהנחה שרבים שמעו שוב ושוב את הביטוי Claude Managed Agents, אך עדיין לא ברור להם מה זה בפועל. המטרה שלו היא להראות איך משתמשים ב-API כדי לבנות מוצר סוכני שניתן, לפחות עקרונית, לקחת לפרודקשן: אפליקציית Deal Desk שמסייעת להעריך חברות פיקטיביות לצורך השקעה או רכישה, תוך שימוש בסוכנים מרובים, MCP, זיכרון, outcomes וזרם אירועים חי.

מהו Claude Managed Agents במונחים מעשיים

בתחילת הסשן מוגדר Managed Agents כאוסף endpoints שמספקים גישה לסוכנים מוכנים להרצה בקנה מידה, עם primitive-ים שניתן לבחור מהם לפי הצורך. הדובר מפרק את המערכת לארבעה חלקים עיקריים. הראשון הוא agent definition: prompt מערכת, מודל, כלים, MCP servers, הרשאות ותתי-סוכנים אפשריים. השני הוא environment: הסביבה שבה הסוכן פועל, כולל קונטיינרים, חבילות, קבצים, רשת ו-sandboxes. השלישי הוא session: השיחה או המשימה המתמשכת, שיכולה לכלול repositories, קבצים ומשתמשים. הרביעי הוא events: כל מה שנשלח לסוכן וכל מה שהסוכן עושה בחזרה.

המסר למפתחים ברור: אל תחשבו על סוכן כקריאה אחת למודל. סוכן פרודקשן הוא תהליך חי. הוא מתחיל, מקבל אירועים, מחזיר אירועים, משתמש בכלים, משנה מצב, לפעמים טועה, לפעמים צריך interruption, ולפעמים צריך לעבוד עם סוכנים אחרים.

סוגי האירועים שמרכיבים את המערכת

הסדנה מפרטת את שכבת האירועים. User events יכולים להיות הודעות, תמונות, מסמכים, interrupts, תוצאות כלים שמורצים אצל המפתח, ואישורי human-in-the-loop לכלים שמורצים בצד Anthropic. Outcomes הם מקרה חשוב במיוחד: המפתח יכול לשלוח הגדרה או rubric, ו-Claude ינסה לבצע את המשימה, לבדוק את עצמו מול הקריטריונים, ולחזור על העבודה עד שהפלט עומד ביעד. Agent events מתארים את מה ש-Claude עושה: תשובות, compaction, tool calls, MCP וכלי ברירת מחדל, וגם coordination events כאשר הוא מפעיל סוכנים נוספים. Session events מתארים סטטוס, שגיאות, retries ו-idle. Span events מסמנים פעולות ארוכות כדי שהמוצר לא ייראה תקוע.

מה בונים בדמו: Deal Desk מרובה סוכנים

הדמו בנוי סביב מוצר פיקטיבי בשם Deal Desk. אפשר לדמיין חברה שמבצעת בדיקות השקעה או רכישה ורוצה לסכם מידע שמפוזר ב-Linear, בקבצים ובמקורות פרטיים. במקום סוכן אחד שמנסה לעשות הכול, המערכת משתמשת ב-multi-agent: סוכן אחד יכול להתמחות במגמות מאקרו, אחר בניתוח פיננסי, אחר במידע תפעולי, ואחר בביקורת. לכל אחד יש context משלו, והסוכן הראשי מתאם ביניהם.

הדובר מציג תחילה גרסה מוכנה של האפליקציה: ממשק צ'אט בסיסי, רשימת סשנים בצד, מידע על הסשן בצד השני, והיסטוריית שיחה במרכז. לאחר מכן הוא חוזר לגרסת starter שבה חלקים מהקוד עדיין לא ממומשים. הצעד הראשון הוא מימוש endpoint שמציג את רשימת הסשנים באמצעות Anthropic SDK. לאחר מכן הוא מממש שליפת פרטי סשן. אלה צעדים פשוטים יחסית, אך הם ממחישים את העיקרון: האפליקציה היא שכבת מוצר סביב APIs של סשנים, אירועים ומשאבים.

Claude בונה את Claude

אחד הרגעים המעניינים הוא כשהדובר מפסיק לכתוב חלק מהקוד ידנית ומבקש מ-Claude Code להשלים את ה-todo-ים הקשורים ל-streaming events ולשליחת events. הוא מציין ש-Claude Code מגיע עם Claude API skill, שמכיר את ה-APIs של Anthropic ובפרט Managed Agents. זו הדגמה של לולאה שמפתחים רבים יכירו: להשתמש ב-Claude כדי לבנות אינטגרציה עם Claude. גם כאשר Claude “מתעצל” ולא משלים הכול בפעם הראשונה, הדובר פשוט מבקש ממנו להמשיך. זה לא רק קטע משעשע; זה מראה איך התיעוד, skills וה-API משתלבים ל-workflow פיתוח חדש.

קונסול, vaults וזיכרון

בזמן ש-Claude Code עובד, הדובר עובר על התיעוד ועל הקונסול. הוא מציג endpoints ל-agents, environments, sessions, events, משאבים, multi-agent threads, credential vaults ו-memory stores. Credential vaults מאפשרים לשמור אסימוני גישה ל-MCP servers בצורה מאובטחת כך ש-Claude יכול להשתמש בכלים בלי שהסודות עצמם ייכנסו לחלון ההקשר. Memory stores מאפשרים לסוכן לקרוא ולכתוב זיכרונות לאורך זמן, כך שסשנים עתידיים משתפרים בזכות ניסיון קודם.

בהמשך, אחרי שהאפליקציה עובדת, הוא יוצר סשן חדש, בוחר Linear MCP וזיכרון, ושולח outcome מורכב: להשוות בין כמה חברות פיקטיביות, לאסוף מידע, לבקר את העבודה, ולוודא שהמסקנות עומדות ב-rubric. הסוכן מתחיל להשתמש בכלים, לקרוא קבצים, ולהפעיל תתי-סוכנים. בקונסול אפשר לראות בזמן אמת את ארבעת הסוכנים שנוצרו, כל אחד עם context משלו, את חיפושי ה-web, את inputs וה-outputs, ואת משך הזמן של פעולות שונות.

מה הודגם בפועל

הדמו מדגים שלושה דברים חשובים. ראשית, Managed Agents מאפשרים לבנות אפליקציה סוכנית מבלי להקים בעצמכם את כל התשתית: agent loop, אחסון durable, streaming, recovery, sandboxing, MCP ו-auth. שנית, multi-agent אינו רעיון תיאורטי; הוא מוצג כ-events וקווים אופקיים בקונסול, עם עבודה מקבילה ותיאום. שלישית, observability הוא הכרח. אם סוכן מקבל משימות משמעותיות, צריך לראות מה הוא עושה, איזה כלי איטי, איפה הוא טעה, ואיך הזיכרון מתעדכן.

המשמעות למפתחים ולסטארטאפים

למפתחים, הסשן נותן תבנית עבודה: התחילו מ-workflow אמיתי, הגדירו agent, סביבה וסשנים, חברו MCP וכלים פרטיים, הוסיפו outcomes כדי שהסוכן ימדוד את עצמו, והשתמשו בקונסול כדי להבין מה קורה. לסטארטאפים, המשמעות היא שאפשר לבנות מוצרים סוכניים מורכבים מהר יותר, בתנאי שמבינים את האינטגרציה ולא מסתפקים בצ'אט. היתרון התחרותי יהיה פחות “יש לנו מודל” ויותר “יש לנו workflow, נתונים, זיכרון, הרשאות ותצפיתיות שמתאימים לדומיין”.

נקודות לקחת

  • Managed Agents הוא סט APIs לבניית סוכנים בני-הרצה, לא רק demo של מודל.
  • סוכן פרודקשן צריך סשנים, events, streaming, כלים, סביבה וזיכרון.
  • Outcomes מאפשרים לסוכן לעבוד מול rubric ולא רק לענות תשובה חד-פעמית.
  • Multi-agent מאפשר חלוקת עבודה בין סוכנים עם context נפרד ותיאום מרכזי.
  • הקונסול, credential vaults ו-memory stores הם חלק מהתשתית שמאפשרת אמינות ואבטחה.
