הבעיה: כשנותנים לסוכן AI את המפתחות לממלכה

המהפכה של סוכני קוד AI — כלים כמו Claude Code, OpenAI Codex CLI, Amp ו-Gemini CLI — שינתה את האופן שבו מפתחים כותבים תוכנה. במקום לכתוב כל שורת קוד ידנית, אפשר לתת לסוכן משימה שלמה: "תבנה לי API של הזמנות", "תתקן את הבאג הזה", "תעשה רפקטורינג למודול הזה". הסוכן קורא קבצים, כותב קוד, מריץ פקודות טרמינל, ומבצע git commit — הכל בעצמו.

אבל יש בעיה מהותית. כשסוכן AI רץ על המחשב שלך, הוא יורש את כל ההרשאות שלך. אותו סוכן שיכול לערוך קבצים בפרויקט שלך, יכול גם לקרוא את מפתחות ה-SSH שלך, לגשת לקרדנשיאלס של AWS, לעיין בקבצים אישיים, או להריץ פקודות הרסניות. ו-LLM-ים הם הסתברותיים מטבעם — הם לא מכונה דטרמיניסטית שתמיד עושה בדיוק מה שאמרו לה.

"LLMs are probabilistic — 1% chance of disaster makes it a matter of when, not if."

דמיינו סוכן שמקבל prompt injection דרך קובץ README זדוני, או שפשוט "מהזה" ומריץ rm -rf ~ במקום לנקות רק את תיקיית build. זה לא תרחיש תיאורטי — זו שאלה של מתי. וככל שיותר מפתחים מריצים סוכנים במצב "YOLO" (עם דגלים כמו --dangerously-skip-permissions ב-Claude Code, או --dangerously-bypass-approvals-and-sandbox ב-Codex), הסיכון הולך וגדל.

הפרדוקס ברור: כדי שסוכני AI יהיו באמת שימושיים, הם צריכים גישה רחבה למערכת הקבצים ולטרמינל. אבל גישה רחבה בלי בקרה היא מתכון לאסון. עד עכשיו, הבחירה הייתה בינארית — או לאשר כל פעולה ידנית (מה שהורג את הפרודוקטיביות), או לתת גישה מלאה ולקוות לטוב.

הפתרון: Agent Safehouse — Deny-First ברמת הקרנל

Agent Safehouse הוא כלי קוד פתוח שנוצר על ידי Eugene Golderman, שמציע תשובה אלגנטית ומעשית לבעיה הזו. במקום לבנות מנגנון אבטחה חדש מאפס, הוא ממנף יכולת מובנית של macOS שרוב המפתחים לא מכירים: sandbox-exec.

sandbox-exec הוא מנגנון sandboxing ברמת הקרנל של macOS (חלק מ-Seatbelt, מערכת האבטחה של Apple). הוא מאפשר להריץ תהליך עם פרופיל הרשאות מותאם אישית — ברמת הקרנל, לא ברמת האפליקציה. כלומר, אפילו אם הסוכן מנסה לגשת לקובץ אסור, הקרנל עצמו חוסם את הגישה לפני שהתהליך בכלל רואה את הנתונים.

העיקרון המנחה הוא Deny-First: כברירת מחדל, שום דבר לא נגיש. רק מה שהוגדר במפורש כמותר — מותר. זה הפוך ממודל ההרשאות הרגיל, שבו תהליך שרץ תחת המשתמש שלך יכול לגשת לכל מה שאתה יכול.

איך זה עובד בפועל

ההתקנה היא פשוטה עד כדי מפתיע. Agent Safehouse הוא סקריפט Bash יחיד, ללא תלויות חיצוניות — רק Bash ו-macOS:

# הורדת safehouse
mkdir -p ~/.local/bin
curl -fsSL https://raw.githubusercontent.com/eugene1g/agent-safehouse/main/dist/safehouse.sh \
 -o ~/.local/bin/safehouse
chmod +x ~/.local/bin/safehouse

# הרצת סוכן בתוך ארגז החול
cd ~/projects/my-app
safehouse claude --dangerously-skip-permissions

כשמריצים סוכן דרך safehouse, הכלי אוטומטית נותן לסוכן הרשאות קריאה וכתיבה לתיקייה הנוכחית (או ה-git root), והרשאות קריאה ל-toolchains מותקנים (כמו Node.js, Python וכו'). כל שאר תיקיית הבית — מפתחות SSH, קרדנשיאלס, ריפוזיטוריז אחרים, קבצים אישיים — חסומים ברמת הקרנל.

רוצים הוכחה? נסו להריץ בתוך ה-sandbox:

# ניסיון לקרוא מפתח SSH — נחסם ע"י הקרנל
safehouse cat ~/.ssh/id_ed25519
# cat: /Users/you/.ssh/id_ed25519: Operation not permitted

# ניסיון לרשום תיקייה של פרויקט אחר — בלתי נראה
safehouse ls ~/other-project
# ls: /Users/you/other-project: Operation not permitted

# אבל הפרויקט הנוכחי עובד כרגיל
safehouse ls .
# README.md src/ package.json ...

תמיכה בכל הסוכנים המובילים

Agent Safehouse תוכנן לעבוד עם כל סוכני הקוד המובילים של 2025-2026:

  • Claude Code (Anthropic) — עם --dangerously-skip-permissions
  • Codex CLI (OpenAI) — עם --dangerously-bypass-approvals-and-sandbox
  • Amp (Sourcegraph) — עם --dangerously-allow-all
  • Gemini CLI (Google) — עם --yolo

שימו לב לדפוס: כל סוכן מציע דגל "מצב מסוכן" שמוותר על אישורים ידניים. שמות הדגלים עצמם (dangerously-*, --yolo) מעידים על כך שגם היצרנים יודעים שזו פשרה מסוכנת. Agent Safehouse מאפשר להשתמש בדגלים האלה בבטחה — כי גם אם הסוכן עושה מה שהוא רוצה, הקרנל מגביל אותו למה שאישרתם.

Shell Functions — אבטחה אוטומטית ושקופה

אחד הרעיונות החכמים ב-Safehouse הוא השימוש ב-shell functions שעוטפים כל סוכן אוטומטית. במקום לזכור להריץ safehouse claude כל פעם, מגדירים פונקציות ב-.zshrc:

# ~/.zshrc
safe() { safehouse --add-dirs-ro=~/mywork "$@"; }

# כל סוכן רץ אוטומטית בתוך ארגז חול
claude() { safe claude --dangerously-skip-permissions "$@"; }
codex() { safe codex --dangerously-bypass-approvals-and-sandbox "$@"; }
amp() { safe amp --dangerously-allow-all "$@"; }
gemini() { NO_BROWSER=true safe gemini --yolo "$@"; }

# צריך להריץ בלי sandbox? השתמש ב-command
# command claude

מהרגע שהפונקציות מוגדרות, כל פעם שמקלידים claude בטרמינל, הסוכן רץ אוטומטית בתוך ארגז החול. אם צריך להריץ בלי sandbox (למשל, לשיחה אינטראקטיבית רגילה), משתמשים ב-command claude כדי לעקוף את הפונקציה.

הגישה הטכנית: למה sandbox-exec ולא Docker

אפשר לשאול: למה לא להשתמש ב-Docker? או ב-VM? התשובה נוגעת לפרקטיקה של פיתוח יומיומי.

Docker מבודד באופן מלא, אבל יוצר חיכוך עצום: צריך לבנות image, להעביר קבצים פנימה והחוצה, לנהל volumes, ולהתמודד עם הבדלי מערכת הפעלה. סוכן AI שרץ בתוך Docker לא רואה את ה-toolchain המקומי שלך, לא יכול להריץ git push בצורה טבעית, ולא מרגיש כמו חלק מסביבת הפיתוח.

sandbox-exec, לעומת זאת, מריץ את התהליך ישירות על macOS, עם אותו kernel, אותם כלים, אותה מערכת קבצים — רק עם מסנן הרשאות ברמת הקרנל. התוצאה: סוכן שמרגיש בדיוק כמו רגיל, אבל לא יכול לצאת מהגדרות הגישה שהגדרתם. אפס חיכוך, אפס תלויות, אפס זמן הקמה.

חשוב לציין שהפרויקט מגדיר את עצמו בצורה כנה כ-"hardening layer, not a perfect security boundary against a determined attacker". זו שכבת הקשחה, לא גבול אבטחה מושלם. נגד סוכן AI רגיל שפשוט שוגה או מקבל prompt injection — זה יותר מספיק. נגד תוקף מתוחכם שבהכוונה מנסה לפרוץ sandbox — זה לא מה שהכלי מבטיח.

יצירת פרופילים מותאמים עם LLM

תכונה מעניינת במיוחד היא היכולת להשתמש ב-LLM כדי ליצור פרופיל sandbox מותאם אישית. הפרויקט מספק prompt מוכן שאפשר להעתיק ולהדביק לכל סוכן AI, שמנחה את הסוכן לבדוק את מבנה תיקיית הבית שלך, להבין אילו כלים מותקנים, ולייצר פרופיל sandbox-exec ב-least-privilege — הרשאות מינימליות בלבד.

ה-LLM גם ידע להציע מיקום עמיד לפרופיל (כמו ~/.config/sandbox-exec.profile), wrapper שנותן גישה לתיקייה הנוכחית, וקיצורי shell לסוכנים המועדפים. זה מטא-שימוש מרתק: להשתמש ב-AI כדי לאבטח AI.

הגדרות ספציפיות למכונה

למפתחים שעובדים עם ריפוזיטוריז משותפים, Safehouse מציע מנגנון אלגנטי להפרדה בין הגדרות משותפות להגדרות מקומיות. באמצעות SAFEHOUSE_APPEND_PROFILE, אפשר להוסיף override-ים מקומיים שלא נכנסים ל-git:

# ~/.zshrc
export SAFEHOUSE_APPEND_PROFILE="$HOME/.config/agent-safehouse/local-overrides.sb"

safe() {
  safehouse \
    --add-dirs-ro="$HOME/server" \
    --append-profile="$SAFEHOUSE_APPEND_PROFILE" \
    "$@"
}

קובץ ה-override המקומי יכול להכיל הרשאות ספציפיות למכונה — למשל, גישת קריאה לתיקיית media של CleanShot או לכונן רשת משותף. זה חשוב בצוותים שבהם כל מפתח עובד עם mount points שונים.

הפילוסופיה: Least Privilege מעשי

הפילוסופיה של Agent Safehouse מסוכמת בארבעה עקרונות:

  • התחל מ-deny-all — כברירת מחדל, הכל חסום
  • אפשר רק מה שהסוכן צריך — כדי לעשות עבודה שימושית
  • שמור על פרודוקטיביות — אבטחה שלא ניתנת לשימוש לא תיושם
  • הפוך את הפחתת הסיכון לקלה כברירת מחדל — אם זה קל, אנשים יעשו את זה

העיקרון האחרון הוא אולי החשוב מכולם. כלי אבטחה שדורש הגדרה מורכבת לא ייושם. Agent Safehouse דורש שורה אחת התקנה ושורה אחת הרצה. זו הסיבה שהוא עובד.

ההשלכות הרחבות: עתיד אבטחת סוכני AI

Agent Safehouse הוא חלק מטרנד גדול יותר בתעשייה: ההכרה שסוכני AI צריכים מודל אבטחה משלהם, שונה מזה של אפליקציות רגילות.

במודל המסורתי, תוכנה עושה בדיוק מה שכתבו לה לעשות. אם יש באג, זה באג בקוד שנכתב. עם סוכני AI, המצב שונה לחלוטין: הסוכן מקבל משימה בשפה טבעית ומחליט בעצמו אילו פעולות לבצע. ההחלטות שלו הן הסתברותיות — לפעמים הוא עושה דברים מפתיעים, לא תמיד במובן הטוב.

זה יוצר קטגוריה חדשה של סיכוני אבטחה:

  • שגיאות לא דטרמיניסטיות — הסוכן יכול לפרש משימה אחרת בכל הרצה
  • Prompt injection — קוד זדוני בקבצים יכול לגרום לסוכן לבצע פעולות לא מכוונות
  • הסלמת הרשאות לא מכוונת — סוכן שמנסה "לעזור" על ידי גישה לקבצים שלא היה צריך
  • דליפת מידע — סוכן ששולח תוכן קבצים רגישים ל-API חיצוני כחלק מה-context

הפתרון של Safehouse — הגבלה ברמת הקרנל — הוא הגישה הנכונה כי הוא לא סומך על הסוכן בכלל. לא משנה מה הסוכן מנסה לעשות, הקרנל חוסם גישה לא מורשית. זה defense in depth אמיתי.

מגבלות ונקודות לשיפור

חשוב להיות כנים לגבי מגבלות הגישה:

  • macOS בלבד — sandbox-exec הוא מנגנון ספציפי ל-macOS. מפתחי Linux יצטרכו פתרונות אחרים (כמו bubblewrap או seccomp)
  • לא חסם מושלם — כפי שהפרויקט עצמו מציין, זו שכבת הקשחה, לא boundary אבטחה מוחלט
  • sandbox-exec מיושן — Apple סימנה את sandbox-exec כ-deprecated, אם כי הוא עדיין עובד ב-macOS הנוכחי. עתיד הכלי תלוי בהחלטות של Apple
  • Network access — ניהול גישת רשת דורש הגדרה נוספת מעבר להרשאות קבצים

למרות המגבלות האלה, היחס עלות-תועלת הוא מצוין. התקנה של 30 שניות מספקת שכבת הגנה משמעותית נגד רוב התרחישים הנפוצים של נזק מסוכן AI.

סיכום: חובה לכל מי שמריץ סוכני AI מקומיים

Agent Safehouse עונה על צורך אמיתי ומיידי. עם הגידול האקספוננציאלי בשימוש בסוכני קוד AI, השאלה אינה האם סוכן ישגה באופן הרסני — אלא מתי. הכלי מציע מענה פרקטי, קל להתקנה, ומבוסס על מנגנון קרנל מוכח.

אם אתם מפתחים על macOS ומשתמשים בסוכני AI כמו Claude Code, Codex, Amp או Gemini — אין סיבה טובה לא להתקין את Safehouse. זה כמו חגורת בטיחות: מקווים שלא תצטרכו אותה, אבל כשתצטרכו — תהיו שמחים שחגרתם.
